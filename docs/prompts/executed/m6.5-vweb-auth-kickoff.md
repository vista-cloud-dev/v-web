# VSL/MSL M6.5 ‚Äî `vweb`: auth middleware ‚Üí DUZ/#200

M6.4 mounted the first real route: `GET /Patient/:id` reads PATIENT (#2) through
VSLFS and returns a minimal FHIR R4 Patient ‚Äî but it is **open** (no auth).
**M6.5 closes it**: an auth middleware that turns an inbound credential into a
VistA principal (DUZ ‚Üí NEW PERSON #200) and rejects the unauthenticated with
**401**. `/healthz` stays open (liveness).

## Where we are (snapshot ‚Äî read, don't re-derive)

- **M6.4 DONE, dual-engine green** (branch `m6.4-fhir-patient`, see
  `docs/memory/m6.4-vweb-fhir-patient.md`): **`VWEBR`** owns the route table
  (`routes(.SRV)` ‚Üí `GET /healthz` + `GET /Patient/:id` ‚Üí `getPatient^VWEBR`) and
  `getPatient` is a clean STDHTTPD handler (`do @ref(.REQ,.RSP)`) that is
  **auth-agnostic** ‚Äî it reads `REQ`, writes `RSP`, and never looks at identity.
  That seam is deliberate: M6.5 adds identity *around* it, not *inside* it.
  Verified bare YDB+IRIS 38/38, vehu 35/35 (live #2 read), foia 27/27;
  `make check` 56/56. MSL pin **v0.12.2**.
- **The middleware seam already exists in STDHTTPD** (consume it, don't build it):
  `$$use^STDHTTPD(.SRV,mwref,.opts)` registers a middleware run **before** the
  handler as `do @mwref(.REQ,.RSP,.CTX)`; a middleware short-circuits by setting
  `CTX("stop")=1` and populating `RSP` (then the handler never runs and dispatch
  returns `RSP("status")`). `$$dispatch` runs the whole chain in registration
  order. So the auth check is a middleware, registered by `routes^VWEBR` (or a
  new `VWEBA`-owned registrar).
- **The DUZ/#200 side already exists in v-stdlib `VSLSEC`** (v‚Üív): `$$duz()` (the
  ambient principal `+$G(DUZ)`), `$$user(duz)` (#200 NAME via VSLFS), `$$hasKey
  (key,duz)`. Use these to resolve/represent the authenticated principal ‚Äî never
  a direct `^VA(200,` / `^XUSEC(` read (waterline).
- **TOKEN VALIDATION DOES NOT EXIST YET anywhere in the stack** ‚Äî there is no
  JWT/OAuth/introspection/token-store module in m-stdlib or v-stdlib. This is
  M6.5's central design decision (see Open questions). Outbound HTTP (for RFC
  7662 introspection) would go through the MSL `STDHTTP`/`VSLIO` client seam, not
  a hand-rolled socket; config (introspection URL, etc.) through `VWEBCFG`/XPAR.

## THIS SESSION (M6.5) ‚Äî `VWEBA` auth middleware

One repo: **v-web** (branch off `main`, e.g. `m6.5-auth`). Per spec ¬ß7:

1. **`VWEBA`** ‚Äî the auth middleware + principal resolution. `authn^VWEBA(.REQ,
   .RSP,.CTX)` (the STDHTTPD 3-arg middleware signature): extract the bearer
   credential from `REQ` (the `Authorization: Bearer <token>` header ‚Äî already
   parsed into `REQ("hdr",...)` by STDHTTPMSG); validate it (see Q1); on success
   resolve the principal (DUZ + #200 NAME via VSLSEC) and stash it on `REQ`
   (e.g. `REQ("auth","duz")`) so downstream handlers/audit can read it; on
   failure set `CTX("stop")=1`, build a **401** (FHIR `OperationOutcome` +
   `WWW-Authenticate: Bearer`), and return.
2. **`/healthz` stays open.** STDHTTPD middleware is **global** (runs for every
   request, before route-match), so `authn^VWEBA` must itself skip the liveness
   path (e.g. allow `REQ("path")="/healthz"` through without a token). Decide and
   document the open-path policy (a small allow-list in VWEBA, or a path prefix).
3. **Register it** ‚Äî `routes^VWEBR` (or a `VWEBA` registrar called from it) adds
   `do use^STDHTTPD(.SRV,"authn^VWEBA")` so it runs ahead of `getPatient`.
   `getPatient` stays byte-for-byte auth-agnostic.
4. **Tests (TDD red-first):** drive the chain via `$$dispatch^STDHTTPD` (fake
   transport, both bare engines): no/!bad token on `/Patient/:id` ‚Üí **401**
   (+ OperationOutcome); a valid token ‚Üí the handler runs (200/404 as M6.4);
   `/healthz` ‚Üí **200 without a token**. Plus the full socket vertical via
   `$$serveConn^VWEBL` (401 on the wire). Token validation that needs a live
   VistA / an auth server is `$text`/config-gated and soft-skips on a bare
   engine (the M6.4 pattern); the principal-resolution unit (VSLSEC) is live-gated.
5. **KIDS** ‚Äî add `VWEBA` to `kids/vweb.build.json` (‚Üí `VWEB*1.0*3`); `make kids`;
   `make check` green. Config params (introspection URL / token policy) as XPAR
   defs if needed.

## Open questions (decide at kickoff ‚Äî Q1 is the big one)

- **Q1 ‚Äî what validates the bearer token?** Pick one, build the seam so the
  others can follow:
  - (a) **RFC 7662 introspection** to a configured auth server (outbound
    `STDHTTP`/`VSLIO`, URL via XPAR) ‚Äî matches the title's "introspection"; needs
    a test double for the dual-engine bare path.
  - (b) **A local token‚Üí#200 store** (a VistA file via VSLFS, or a VSL token
    module) ‚Äî fully self-contained, dual-engine, no outbound dependency.
  - (c) **Kernel's existing secure-token / Application-Proxy mechanism** ‚Äî most
    "VistA-native" but the most coupling; verify a Supported API exists (vdocs)
    before committing.
  Recommendation: start with the **seam + a local/double-backed validator** so
  M6.5 lands dual-engine-green, and wire real introspection behind the same seam
  as a follow-up (mirrors the HTTP-first / TLS-gap-loud posture).
- **Q2 ‚Äî does establishing the principal mean setting `DUZ`?** VSLSEC reads the
  *ambient* `DUZ`. Decide whether `authn^VWEBA` sets `DUZ` (so VSLFS/Kernel calls
  run as the user) or only annotates `REQ`. Setting `DUZ` has process-scope
  implications under the serial worker loop ‚Äî document it.
- **Q3 ‚Äî audit.** Spec ¬ß7/¬ß12: emit an auth event to the injected log hook
  (`opts("logref")`)? Keep it minimal for M6.5.

## Decisions / guards (do not re-litigate)

- **Consume the seams** ‚Äî middleware via `$$use`/`$$dispatch^STDHTTPD`; principal
  via `VSLSEC`; outbound HTTP (if any) via `STDHTTP`/`VSLIO`; config via
  `VWEBCFG`/XPAR. Never hand-roll a token parser's transport or a #200 read.
- **Waterline holds** ‚Äî `VWEBA` is layer `v`; no direct `^VA(200`/`^XUSEC`/
  `^XUSER` (use VSLSEC, v‚Üív); v‚Üím for STD*. Keep the ICR registry honest (a real
  Supported-API marker if you touch a Kernel auth API; `@icr DBS`-style notional
  marker otherwise ‚Äî see [[notional-dbia-not-a-blocker]]).
- **Handler stays auth-agnostic** ‚Äî identity lives in the middleware, not in
  `getPatient`. `/healthz` stays open.
- **Engine access = driver stack only** ‚Äî `m test --engine ‚Ä¶ --docker ‚Ä¶`; live
  on **vehu** + **foia-t12**; NEVER raw docker exec.
